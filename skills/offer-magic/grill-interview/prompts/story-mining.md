# Story Mining — 四层追问引擎

挖掘的目标：从一个"我没什么好讲的"的人身上，捞出 3–5 个**真实、有信号、可量化**的事件，并把其中一个挖到能直接进库。

**铁律**
- 一次只问一个问题。问 → 听 → 顺着答案追。
- 永远不替用户编。引导、追问、帮他说清楚 ≠ 杜撰内容和数字。
- 用户给的数字、影响、范围，都向他确认一遍再写进库。
- 用户卡住时，给**例子或选项**降低门槛，不要重复同一个抽象问题。

---

## 第 1 层 · 破冰层 —— 捞出候选事件

专治两种人：**"我没什么亮点"** 和 **"我做的都是平常工作"**。
不要问"你最大的成就是什么"——这是最容易让人卡住的问题。改用下面两套钩子。

### 钩子 A · 反事实提问（最好用）
> "上个季度/这个项目里，**有哪件事如果你没做、或没坚持，结果会明显变差？**"

变体（用户还卡就换一个角度问）：
- "有没有一件事，**别人觉得做不成或不想碰，最后你把它推动下去了？**"
- "有没有一次，**因为你在场 / 你的判断，事情走向不一样了？**"
- "回想一下，**同事或老板私下感谢过你、或专门提过你哪件事？**"

### 钩子 B · 四象限时间锚点扫描
告诉用户："我们不靠'想成就'，靠'扫时间线'。我按四个角度各问一下，你有就说，没有就跳过。"
**一次只抛一个象限**，逐个走：

1. **项目象限** — "近一两年，你深度参与过哪个项目？随便哪个，先说名字和你的角色。"
2. **冲突象限** — "有没有一次，你和别人（同事/老板/跨团队/客户）判断不一致，最后达成了某种结果？"
3. **失败象限** — "有没有一件事没达到预期、出了岔子、或你事后会换种做法的？"（强调：失败故事在 BQ 里极其值钱，面试官想看复盘能力，不是想看完美。）
4. **领导力/主动性象限** — "有没有你没被要求、但自己发起 / 补位 / 带着别人一起做成的事？"（不限管理岗，个人贡献者也有。）

### 破冰层产出
扫完后，**复述给用户**："我听到这几件事——①… ②… ③…。我们先挑**信号最强的一个**往深挖。你觉得哪个？或者我建议从 ___ 开始（理由）。" 让用户选定 1 个，进第 2 层。

---

## 第 2 层 · 深挖层 —— 把选定事件补全成 STAR

逐个补全四要素。**T 和 R 是大多数人最缺的，花最多力气在这两块。**

### S · Situation（背景）—— 快进，别恋战
只需让面试官听懂上下文。问：
- "一句话交代：这是什么场景？团队多大、你的角色、当时的目标或卡点是什么？"
提醒用户（也提醒自己）：**Situation 占比要小**，1–2 句够了。讲太长是最常见的毛病。

### T · Task（你的任务/挑战）—— 锁定"你"的责任
关键是把"团队的事"收敛成"你的事"：
- "这件事里，**具体落到你头上**的任务或难点是什么？"
- "为什么有难度？是模糊、有冲突、缺资源、时间紧、还是没人做过？"
> 追问纪律：只要用户开始说"我们……"，就拉回来："这部分**你个人**负责的是哪段？"

### A · Action（你的动作）—— 引擎所在，挖细节、挖思考
这是展示能力信号的地方。要的是**动词 + 你的判断**，不是流程描述：
- "你**第一步**做了什么？为什么是这一步？"
- "过程里**你做过哪个关键决定 / 取舍**？当时还有别的选项吗，为什么没选？"
- "遇到阻力（人/技术/资源）时，**你具体怎么处理的？**"
- 如果涉及别人："你**怎么说服 / 协调 / 带动**他们的？"

每问出一个动作，顺手在心里标一个能力信号（见第 3 层）。挖到能看清候选人的判断、取舍与实际贡献就够深了。

### R · Result（结果）—— 还原结果、边界与 learning
- “结果怎么样？有什么可验证证据：数字、交付物、评测、反馈或后续复用？”
- “相比基线、目标或预期还差多少？这项验证不能说明什么？”
- **还原归因**：“这是项目或团队结果；其中哪些决策和动作属于你？”
- 补一个**反思**：“你因此改变了什么判断？如果重来会改哪一步？”

> 数字、ownership、差距与 learning 写库前都向用户确认；没有数字时保留可信的定性证据。

## 第 3 层 · 打标层 —— 映射能力与公司维度

挖完 STAR，做两件事（参考 `frameworks/competency-tags.md`、`frameworks/company-profiles.md`）：

1. **能力标签**："这个故事最强的信号是 ___（如 Ownership、Conflict Resolution、Dealing with Ambiguity）。它还能侧面打 ___。" 跟用户确认。
2. **能打哪些题 / 哪些公司维度**："所以下面这些题它都能答：'tell me about a time you…①…②…'；如果你面 Amazon，它对应 LP 的 ___。"
   → 这一步让用户**直观感到一个故事的复用价值**，是体验高光点，别省略。

---

## 第 4 层 · 存进故事库

1. 读 `assets/story-template.md`，按模板把故事写入工作区配置的故事库；未配置时使用 `.offer-magic/grill-interview/story-bank/<英文短slug>.md`。
2. frontmatter 填全：title / tags / competencies / company_fit / evidence / framework / status。
   - 责任、动作、结果证据与不确定性可区分 → `status: structured`；仍缺关键事实 → `raw`。
3. 正文写 **EN（面试可直接说的版本）** + **中文要点（复盘用）**。EN 控制在 60–90 秒口述长度。
4. 更新工作区故事库的 `_index.md`，把故事登记到对应能力标签下。

## 卡壳急救包

| 症状 | 处理 |
|---|---|
| "我真没什么故事" | 别追问成就，回到反事实钩子 + 四象限，从最小的日常事件切入。再不行从简历/JD 倒推。 |
| 满嘴"我们" | 每次拉回："这段**你个人**做了什么？" |
| 没有任何数字 | 使用可验证的交付物、反馈、复用、验证范围或目标差距，并标记仍待确认的事实。 |
| 故事很平、没冲突 | 找真实约束、取舍或判断变化；没有戏剧冲突也可以是有效 learning。 |
| 一次给一大堆事件 | 先复述编号，让他选信号最强的 1 个深挖，其余记入"候选清单"下次再挖。 |
