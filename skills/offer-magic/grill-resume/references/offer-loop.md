# Offer Loop Reference

Read this reference only for substantive JD analysis, resume restructuring, or interview-hook design. Skip it for a pure wording correction, CSS adjustment, or artifact rebuild.

## Build one evidence map

Create an internal table with one row per important role signal:

| Role signal | Priority | Evidence strength | Source | Resume placement | Likely follow-up | Risk |
|---|---|---|---|---|---|---|
| Exact JD requirement or inferred need | hard / core / supporting / environment | strong / adjacent / none | code path, test, artifact, user statement | intro, bullet, other information, omit | question an interviewer may ask | exaggeration, gap, or ambiguity |

Rules:

1. `strong` requires a direct, verifiable example. A concrete implementation, test, delivered artifact, or confirmed result counts.
2. `adjacent` means the evidence transfers but is not the same experience. Phrase it as adjacency; do not silently promote it.
3. `none` never becomes a resume keyword merely to improve matching. It is either omitted, disclosed as a gap, or turned into a focused question for the user.
4. Explicit JD text and inferred environment signals remain visibly separate.
5. The map is a reasoning aid, not another repository artifact unless the user asks to keep it.

## Calibrate evidence strength to career stage

Use an adaptive evidence ladder, not a permanent template:

1. **First internship, no prior internship:** let relevant projects, research, coursework, and open-source work carry the resume. Judge scope, decisions, verification, and learning at student-project scale; do not manufacture an Experience section or demand full-time business impact.
2. **Later internship applications:** lead with relevant internships when they provide stronger workplace evidence. Keep projects that add distinct mechanisms, ownership, research depth, or product judgment.
3. **Full-time after professional work begins:** lead with relevant work experience. Retain internships for early-career continuity and projects for exceptional relevance, then compress them as stronger professional evidence accumulates.

This ladder changes evidence priority and proof burden, not truth standards. Projects can prove technical capability at every stage, but do not silently prove production scale, organizational influence, or people management.

## Select the resume claims

Choose claims by this order:

1. Hard-condition proof that prevents immediate rejection.
2. Core technical and product evidence that differentiates this candidate.
3. Results, scale, evaluation coverage, or shipped artifacts.
4. Judgment and trade-offs that create useful follow-up questions.
5. Supporting preferences and generic skills.

For each selected claim, apply four checks:

- **Truth:** Can every noun, number, and ownership verb be defended?
- **Match:** Which role signal does it help prove?
- **Scan:** Can the key point be recovered from the intro, first clause, number, or bold anchor?
- **Follow-up:** Is there a real code path, decision, failure, or artifact behind the likely interview question?

If a claim fails Truth, remove it. If it fails Match and Scan, it should usually yield space. If it passes Follow-up but not Match, keep it only when it supports the candidate’s broader target profile.

Use a contribution boundary for project claims:

- Relevance and defensible learning can justify a project even when its outcome is ordinary or unfinished.
- Attribute only the candidate's decisions and work. Present team, product, competition, or business outcomes as project context unless personal causality is evidenced.
- When useful, pair the result with its baseline, target, remaining gap, or validation limit. A passed check is not proof of broader product or research success.
- Prefer a reusable judgment, failed assumption, or changed approach over heroic language.

## Shape project bullets

Use the smallest structure that preserves causality:

`背景或约束 → 判断或设计 → 机制或行动 → 结果、差距或验证 → learning`

Not every bullet needs all four clauses. The project intro can carry shared background so bullets spend space on decisions, mechanisms, and evidence. Avoid a uniform “action + inflated percentage” formula when the repository truth is better expressed through boundaries, recovery, evaluations, or delivered artifacts.

Build a claim-to-talk map while editing: each project should have a short spoken logic, and each important bullet should be one thesis the candidate can point to and expand with background, actions, evidence, limits, and learning. If the spoken account and resume require different ownership, facts, or chronology, correct the resume before rendering.

Order project bullets for screening value:

1. defining architecture or workflow;
2. differentiating technical mechanism;
3. safety, evaluation, recovery, or quality evidence;
4. delivered result or current state.

Move a stronger verified result earlier when it is the most persuasive signal.

## Predict defensible interview hooks

Convert only high-priority map rows into follow-ups. For each anchor, write an internal answer sketch:

- What problem or constraint existed?
- What decision or trade-off was made?
- Where is that behavior visible in code or artifacts?
- What failed, remained incomplete, or differed from the initial expectation?
- What was learned and what would change next?

The resume does not need to contain all five answers. It should contain enough specificity that the question is natural and enough truth that the candidate can answer it.

## Completion gate

Content is ready to render only when:

- every hard requirement is marked met, adjacent, unmet, or irrelevant to the actual application type;
- every retained number and ownership verb has a source;
- each major experience or project has a clear, non-duplicative role in the candidate story;
- the earliest bullets cover the strongest screening evidence;
- bold anchors include both role-match signals and defensible discussion points;
- likely follow-ups do not expose unsupported claims;
- major bullets map one-to-one to defensible interview evidence and learning;
- language choices follow the workspace profile or the resume’s established language.
