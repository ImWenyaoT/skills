# 简历写作检查

把 bullet 写成可扫描、可追问的论点，不套固定成果公式。

## 先判断是否值得写

优先保留：岗位相关性、可验证贡献、关键判断或机制、结果与边界、可复用 learning。项目成功本身不是个人能力证明；普通或未完成的项目，只要相关且有真实积累，也可以成立。

## 最小表达结构

按事实选用：

`背景或约束 → 本人判断/行动 → 机制 → 结果、差距或验证 → learning`

不要求每条写全。共享背景放在项目简介；bullet 只保留一个论点和核心论据。数字只有在真实、可解释且能提供范围或比较时才优于定性证据。

简历是换取面试的敲门砖，不是项目档案。默认把项目 bullets 控制在 10 条以内；每条先用加粗的“名词论点：”让读者立即知道在证明什么，再给一组最强论据。实现细节、备选方案、失败路径、限制和 learning 留给面试展开。

先确保非技术 HR 能看懂，再考虑技术精度。岗位会检索的术语或无法替代的机制名可以保留，其余黑话改写为“做了什么、为什么这样做、如何保证结果”；如果术语本身需要解释，它通常不应抢占简历主句。

以每条约 48–52 个中文字、视觉上接近“一行一句”为排版诊断，不把它当作压缩指标。单一论点若需要“关键机制 + 验证边界”才能可信，自然占一至两行是合理的；不得为了凑一行删掉使论点成立的核心证据。超过时先删第二论点、重复背景和次级实现细节。

把用户声明的视觉行数视为硬约束。优先形成完整的两行或三行节奏，避免两行半和只有少量文字的孤立尾行；如果在保留核心证据后仍无法压进上限，拆成两个职责不同的 bullet，而不是缩小到不可读字号或删除论据。不要插入手工换行来伪造节奏，必须以最终 PDF 的实际换行为准。

加粗承担扫描导航和岗位匹配，不承担装饰。短论点标签、JD 的高信号机制、数量证据或验证边界可以成为视觉锚点；同一条只保留最有筛选价值的少量锚点，避免整句加粗导致权重失效。

## 归因与验证

- 分开项目/团队结果与个人贡献；逐一核对 ownership 动词。
- 结果尽量说明基线、目标、当前差距或验证边界。
- 通过测试、评测或检查，只证明其覆盖范围内的结果。
- 避免“负责”和空泛形容，也避免用“主导、推动、提升”制造因果。
- 缺事实时删除、标记待确认，或改写为可验证的机制与交付物。

## 简历—面试一致

每条重要 bullet 应能在面试中按同一顺序展开：背景、个人动作、证据、局限、learning。简历是论点加核心论据；口述补充细节。两者若在事实、归因或时间线上不一致，以事实修正简历。

## 其他信息与机筛

有 JD 时，把“其他信息”当作窄而明确的 ATS 关键词桥梁，而不是通用自我评价：

- 先从 JD 摘出与候选人事实相符的工具、机制、方向和工作方式；尽量保留原词与原句结构，让机器能直接匹配。
- 通用版本先比较多份代表性 JD，提取目标赛道反复出现的能力交集；单个岗位只影响定向版本，不应重写稳定的项目事实。
- 当用户指定核心 JD 与泛化辅助 JD 时，先用核心 JD 决定叙事、排序与视觉锚点，再只用辅助 JD 检查通用能力缺口和单一岗位过拟合；辅助语料不得稀释核心方向。
- 先划分岗位分支再提词：Harness、Agent 应用与模型训练即使混在同一批 JD 中，也不能互相借用无证据要求。
- 只做使其符合候选人口吻与事实边界的最小改写；不能证明的职责、资历和技能不摘录。
- 相邻关键词合并进自然句子，不堆同义词，不把项目正文已有的所有术语重复一遍。
- 同时保留至少一条来自候选人真实行为的差异化信号，例如持续使用、直接阅读一手资料、独立验证机制或明确的产品判断。
- 最终同时通过两道检查：机器能找到 JD 原词；HR 能看到证据、强度和区别，而不是复制粘贴。
- 排版出现明显半行或短尾行时，优先重组已有论据；需要增加长度时只使用相邻真实证据，需要缩短时先移除重复背景和面试层实现细节。不得为撑行重复同义词、加入空泛形容或扩大事实边界。

把缺词分成两类：已有直接证据但缺少 JD 原词的是措辞缺口，可做最小改写；只有相邻经验或没有证据的是证据缺口，保留边界并留给未来项目或单独版本。技术栈、产品范围、部署经历和 ownership 不会因加入关键词而变成证据。

## AI-native 项目

AI-native 岗位先证明场景判断和 Agent 机制，再证明产品结果，最后才决定是否暴露 Web 技术栈。每条先说明什么 Context 下出现了什么 AI 问题，再给判断、机制和验证；“用了什么”只有在解释“解决了什么”时才有价值。

把 Harness、Context、Tool、Memory、评测或 Human-in-the-loop 等词放进真实问题与机制中，不建立术语清单，也不在每条 bullet 重复给系统贴 Harness 标签。Model、Agent 与 Harness 只在对应层确实影响设计时出现；模型名、SDK 或 API 只在具有 JD 检索价值或因果解释价值时出现，易过时的版本与底层替代方案留给面试。

候选人保留责任链：问题拆分、实现、测试和最终审查由候选人的判断连接起来，AI Agent 是加速手段而不是代替 ownership 的主语。英文技术词只保留目标 JD 原词或 Dictionary of AI Coding 的规范词条；字典批准词形，不证明实现，也不形成关键词配额。

## Summary

只在用户明确要求时讨论或加入可见 Summary。优先写已经核实、能解除首轮硬门槛的事实，例如毕业状态、到岗时间、实习频率、持续时长或地点；地点等条件事实未经确认保持缺失。Summary 不承担自我评价，也不挤掉更强的项目或科研证据。

## 常规检查

- 倒序排列；应届候选人优先 Projects 与 Education。
- 一页优先，统一时间、标点、字体和语言。
- 使用真实的 JD 词汇，但不堆关键词或写无证据技能。
- 关键信息用可解析文本，不藏在图片、文本框或页眉页脚。
