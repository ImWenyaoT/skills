# Writing Technical Resumes

This skill is the **content layer** of a resume: *what to write and how to phrase it*. Layout,
fonts, and PDF export belong to whatever resume tool you use — keep content and presentation
separate.

**Core principle:** every bullet should be **results-oriented, quantified, strong-verb-led,
non-repetitive, and honest.** Most weak resumes fail because they list tasks ("responsible for
X") instead of outcomes ("did X, which produced Y").

## Five writing techniques (run on every bullet)

1. **End on the hardest quantifiable result.** Put the most striking number/fact at the
   sentence's end. *e.g.* `Coached team through a 15-month cycle; ultimately captured 5% of a
   $130M market`.
2. **Write outcomes, not tasks — never restate.** No two bullets may re-describe the same thing
   in different words. Each bullet carries one of: context, action, or result. Spot a restatement
   → **merge** (that's de-duplication, not deletion).
3. **Layer the quantification: scale → output → value.** *e.g.* `team of 25 … produced 14
   projects … $13M revenue`.
4. **Kill filler verbs.** Replace empty verbs ("responsible for", "involved in", "participated
   in", "worked on", "helped with") with strong action verbs — see [references/verb-list.md](references/verb-list.md).
5. **Name the artifact + external adoption.** A concrete product name plus third-party
   validation. *e.g.* `'C-Tech' system — licensed to a global manufacturer`. "Reused by team X",
   "merged to mainline", "published (DOI)" all count as external anchors.

**Numbers over adjectives, and bold the key ones.** A number is more legible than a word; bold
the decisive figures so a reviewer's eye lands on the data. Aim for each entry to show all three
layers: **what it is / how you did it / quantified result**.

## Bullet structure (STAR logic, no labels)

- Organize with STAR / three-part logic (context → action → result), but **never write the
  labels** "Context:/Action:/Result:" into the bullet — prefixing them looks amateurish.
- **Start every bullet with a strong verb.** Don't repeat a verb *within one entry*; keep
  variety across the whole resume. Core engineering verbs (design/implement/build) will
  naturally recur across projects — that's fine; the real target is filler verbs (technique 4).
- Full method and a worked example library: [references/writing-method.md](references/writing-method.md).

## Tailoring per role / JD

- **Write a mega superset first.** Put every relevant experience into one master source (anything
  you actually did, can explain, and can defend in an interview). Tag each with an impact weight
  and which roles it belongs to.
- **Cut by relevance, not by time.** For a given role/JD, judge each experience against the job
  description's keywords and requirements; **filter by relevance, order by impact.** The reader
  should grasp what you did at a glance.
- **Role flavor:** an engineering role emphasizes the engineering loop (tests/CI/eval/systems); a
  product role emphasizes user value and 0→1 judgment. A newcomer with no usage metrics should
  **not invent numbers** — lean on demonstrated judgment instead.
- **One page:** cut by relevance first; only then lean on font/spacing auto-fit. Never shrink type
  to cram in *irrelevant* content.

## Skills section

- **Only list what you can defend.** Anything on the page is fair game in an interview; a blank
  stare on your own listed skill sinks you. Drop what you're not fluent in.
- **Normalize terminology:** `java → Java`, `mysql → MySQL`, `nextjs → Next.js`, `fastapi → FastAPI`.
- **Order:** languages → frameworks/components → fundamentals → tools. Detail the sub-topics under
  each so an interviewer can question along the lines you wrote. Don't pile up model/product names
  as if they were skills.

## Self-evaluation / summary

- One line = **one judgment + one concrete fact that proves it.** A bare adjective ("loves
  learning") is worthless; back every claim with a real event. Form: `judgment — concrete fact`.
- Draft this section even if the candidate finds it hard to write about themselves — but every
  supporting fact must be verifiably true (never fabricated).

## Honesty red line (overrides every technique above)

- **Never fabricate** a number, experience, or status. If a metric is an old snapshot, label it
  "(snapshot)"; don't inflate.
- Report paper/project status **as it actually is**: published / under review / in progress. Never
  write "submitted" for unsubmitted, "accepted" for not-accepted.
- **Take the lowest honest ownership word:** second author → "co-author / co-built", not
  "led / first-author / sole creator".
- Don't pile model names as skills; an AI resume should frame AI as an **engineerable system**
  (eval / CI / harness / feedback loop), not a list of model names.
